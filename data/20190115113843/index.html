Driving Faster Content Preparation with Scalable Chunked Encoding利用可扩展的分块编码推动更快的内容准备

by Karthikeyan N (Principal Engineer, Media Server Technologies) 作者： Karthikeyan N （媒体服务器技术首席工程师）

Key factors that enable scalability 实现可伸缩性的关键因素

Why chunked encoding?为什么要进行分块编码？

One of the biggest challenges with on-demand video is delivering the most optimal content for distribution.按需视频面临的最大挑战之一是为分发提供最佳内容。 While multiple ABR representations, higher resolutions like 4K, higher frame rates and newer compression formats like HEVC and VP9 contribute to improvements in user experience, the encoding process gets increasingly complex and hence slow.虽然多个ABR表示，诸如4K的更高分辨率，更高的帧速率和更新的压缩格式（如HEVC和VP9）有助于改善用户体验，但编码过程变得越来越复杂并因此变得缓慢。 This puts content preparation workflows under a lot of pressure to deliver content within the stipulated turnaround time.这使得内容准备工作流程面临很大压力，无法在规定的周转时间内交付内容。 With chunked encoding, we can effectively address this concern.使用分块编码，我们可以有效地解决这个问题。

Scale of increase in complexity 复杂程度增加的规模

  • 1080p30 to 4Kp60 – 8x 1080p30至4Kp60 - 8x
  • H.264 to HEVC/VP9 – 3 to5x H.264至HEVC / VP9 - 3至5x
  • Adding all ABR profiles – 2.5 to5x添加所有ABR配置文件 - 2.5到5x

While the process of encoding a single file in 1080p30 typically requires one hour, it can now extend up to several days with all the above changes! 虽然在1080p30中编码单个文件的过程通常需要一个小时，但现在可以将所有上述更改延长至几天！

Since the distribution costs far exceed the encoding costs, it makes business sense to add more compute resources to ensure faster availability of content to subscribers.由于分发成本远远超过编码成本，因此添加更多计算资源以确保更快地向订阅者提供内容具有商业意义。 We can then reduce the turnaround time by fragmenting the source content into multiple chunks, and independently encoding them in parallel across multiple compute resources.然后，我们可以通过将源内容分段为多个块来减少周转时间，并在多个计算资源之间并行地独立编码它们。 This is exactly what THINKode, Ittiam's machine learning based content adaptive encoder, enables.这正是Ittiam基于机器学习的内容自适应编码器THINKode所能实现的。 However, a key parameter that defines the success of our chunked encoding scheme is ensuring scalability over a large number of resources.但是，定义分块编码方案成功的关键参数是确保大量资源的可扩展性。

Chunk size Vs video quality trade-off块大小与视频质量权衡取舍

Fragmenting the source content limits the amount of frames an encoder can use during encoding.对源内容进行分段会限制编码器在编码期间可以使用的帧数。 This presents a trade-off between the chunk size and the quality of encoding.这表示在块大小和编码质量之间进行权衡。 Although our objective is to minimize the chunk size to increase opportunities for parallel processing, the following factors force a lower limit on the chunk size:虽然我们的目标是最小化块大小以增加并行处理的机会，但以下因素会强制块大小的下限：

  • Typical encoding algorithms will not be efficient for chunks that are smaller than a certain threshold due to restricted look ahead processing, resulting in reduced quality.由于受限制的前瞻处理，典型的编码算法对于小于特定阈值的块不是有效的，导致质量降低。
  • Key characteristics of the input video such as encoding GOP size and any scene-cuts will also limit the minimum chunk sizes we can use.输入视频的关键特性（例如编码GOP大小和任何场景切换）也将限制我们可以使用的最小块大小。

To determine the minimum chunk size that delivers acceptable quality, we must understand the typical GOP sizes and scene transitions over a large content set. 要确定提供可接受质量的最小块大小，我们必须了解大型内容集上的典型GOP大小和场景转换。 We also need to experiment with various chunk sizes to assess the overall quality degradation. 我们还需要尝试各种块大小来评估整体质量下降。

Chunk size Vs performance trade-off块大小与性能权衡

Even if we choose a very small chunk size that ensures zero quality degradation, we need to consider several performance factors that limit efficient parallelization.即使我们选择一个非常小的块大小来确保零质量下降，我们也需要考虑几个限制高效并行化的性能因素。 Furthermore, smaller chunk sizes lead to higher system management overheads.此外，较小的块大小会导致较高的系统管理开销。 The chunk size therefore needs to be balanced with the overall turnaround time achievable, using the available storage infrastructure and compute resources.因此，需要使用可用的存储基础架构和计算资源来平衡块大小与可实现的总周转时间。 Take a look at how we can do so based on three main performance factors:根据三个主要性能因素，看看我们如何做到这一点：

1. File storage access speed 1.文件存储访问速度

While we attempt to parallelize encoding across multiple compute resources, the source files typically reside on a single file storage system.当我们尝试跨多个计算资源并行化编码时，源文件通常驻留在单个文件存储系统上。 The question here is, 'does the download speed of each chunk match the processing speed of the encoding process task while downloading multiple chunks of source file in parallel?'这里的问题是，“并行下载多个源文件块时，每个块的下载速度是否与编码处理任务的处理速度相匹配？” If the answer is yes, then the encoding process task does not starve of input data.如果答案是肯定的，则编码处理任务不会使输入数据匮乏。

Even with high speed, distributed public cloud storage services like AmazonS3, the download speed does not scale linearly with the number of parallel downloads beyond a certain point.即使是像AmazonS3这样的高速分布式公共云存储服务，下载速度也不会随着超过某一点的并行下载数量而线性扩展。 And if the content provider has hosted the content on private servers accessed through an external network, the download speeds could be even lower and inconsistent.如果内容提供商已将内容托管在通过外部网络访问的私有服务器上，则下载速度可能更低且不一致。

To reduce wastage of cloud compute resources, we must adopt an optimal chunk size dynamically based on the file storage type and actual download speed available. 为了减少云计算资源的浪费，我们必须根据文件存储类型和可用的实际下载速度动态采用最佳块大小。

2. Shared filesystem access speed 2.共享文件系统访问速度

In a chunked encoding system, a shared filesystem may be needed to store metadata needed across encoding tasks.在分块编码系统中，可能需要共享文件系统来存储跨编码任务所需的元数据。 It also stores the intermediate media/metadata output from each chunk process task required for the stitcher to generate the final output.它还存储从拼接器生成最终输出所需的每个块处理任务的中间媒体/元数据输出。

Amazon provides Elastic File System (EFS) for use as shared filesystem. Amazon提供弹性文件系统（EFS）用作共享文件系统。 The throughput of EFS depends on the size of data stored in the EFS at that point of time. EFS的吞吐量取决于该时间点存储在EFS中的数据大小。 Take a look at the EFS throughput for sample file system sizes under steady state operation:在稳态操作下查看样本文件系统大小的EFS吞吐量：

Filesystem Size (GB)文件系统大小（GB） Baseline Aggregate Throughput (MBps)基线总吞吐量（MBps）
10 10 0.5 0.5
256 256 12.5 12.5

Now, let us consider a typical ABR encoding recipe with 10 representations:现在，让我们考虑具有10个表示的典型ABR编码配方：

  • ~10GB shared filesystem will be required for one hour of content, if the output is generated at a total bitrate of ~2.5MBps如果输出以~2.5MBps的总比特率生成，则需要~10GB共享文件系统一小时的内容
  • 1080p24 input content can be encoded at quality equivalent to x264 two pass encoding at a very slow preset at a speed of ~1.82fps 1080p24输入内容的编码质量相当于x264双通道编码，速度非常慢，速度为~1.82fps
  • The output will thus be generated at {(2.5MBps * 1.82fps) / 24fps} = 0.189MBps因此，输出将以{（2.5MBps * 1.82fps）/ 24fps} = 0.189MBps生成
  • As output is generated for each chunk, the filesystem grows and the throughput increases, allowing us to当为每个块生成输出时，文件系统增长并且吞吐量增加，允许我们
  • process a maximum of three chunks in parallel towards the end of the encoding.在编码结束时并行处理最多三个块。
  • In addition to this, the stitcher can read at an average of only 0.5MBps, once all the chunks are processed.除此之外，一旦处理完所有块，拼接器平均只能读取0.5MBps。

Though the throughput can be increased by storing some persistent data in EFS, this adds to the overall cost of encoding.虽然通过在EFS中存储一些持久性数据可以提高吞吐量，但这会增加编码的总体成本。 Scalability will thus depend on how well we design the encoding workflow to minimize the usage of shared filesystem. 因此，可伸缩性将取决于我们如何设计编码工作流以最小化共享文件系统的使用。

3. Instance management 3.实例管理

The time required to dynamically acquire cloud compute instances and the associated billing cycle, which typically differs across cloud providers, also influence scalability.动态获取云计算实例所需的时间和相关的计费周期（通常因云提供商而异）也会影响可扩展性。 If the objective of scalability includes optimal utilization of acquired resources in addition to reducing the turnaround time of encoding, we also have to ensure efficient scheduling of chunked encoding tasks across the acquired resources.如果可扩展性的目标除了减少编码的周转时间之外还包括获取资源的最佳利用，我们还必须确保在所获取的资源上有效地调度分块编码任务。

For example, Amazon charges for a minimum billing cycle of one hour, in which case, it is cost effective to make use of the complete billing cycle.例如，亚马逊收费的最小计费周期为一小时，在这种情况下，使用完整的结算周期具有成本效益。 Instead of acquiring six resources that perform a 10 minute encoding task each, we can acquire a single resource and schedule the six tasks sequentially – at the cost of increased turnaround time for those tasks.我们可以获取单个资源并按顺序安排六个任务，而不是获取每个执行10分钟编码任务的六个资源，而代价是增加这些任务的周转时间。

Therefore, the best approach is to choose the optimum number of chunks for processing in parallel based on a perfect tradeoff between cost and overall speed gain across all compute resources . 因此，最佳方法是基于所有计算资源之间的成本和总体速度增益之间的完美折衷，选择用于并行处理的最佳块数 。 If the billing cycle or cost is not a constraint, then consider the time it takes to launch an instance to determine the optimum number of chunks for parallel processing.如果计费周期或成本不是约束，则考虑启动实例以确定并行处理的最佳块数所花费的时间。

Gear up for speed加快速度

The impact of the above factors becomes manifold when multiple contents are encoded in parallel.当多个内容并行编码时，上述因素的影响变得多种多样。 However, it is possible to effectively address these challenges by replicating what we have done with THINKode software.但是，通过复制我们对THINKode软件所做的工作，可以有效地解决这些挑战。 This involves implementing schemes that monitor the critical factors and automatically identify the optimum number of chunks to be processed in parallel for efficient use of cloud instances.这涉及实施监控关键因素的方案，并自动识别要并行处理的最佳块数以有效使用云实例。
